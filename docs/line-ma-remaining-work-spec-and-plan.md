# LINE MA 残課題：仕様書・実装計画（I3 除外）

本書は、既にリポジトリへ実装済みの **Phase A〜I（I3 を除く）** を前提とし、**プロダクトとして運用可能にするための残課題**を整理する。外部 CRM／スプレッドシート連携（**I3**）はスコープ外とする。

---

## 1. 背景と目的

### 1.1 現状

- **API・バッチ・DB マイグレーション**は一通り揃っている。
- 一方で、**管理 UI の網羅**、**本番運用の手順の固定**、**品質ゲート（Lint 等）**、**予約ドメインの高度化**などは、計画段階で MVP に留めた部分やインフラ依存が残っている。

### 1.2 本書の目的

| 文書 | 内容 |
|------|------|
| **仕様書** | 何を満たせば「課題解決済み」とみなすか（要件・受け入れ基準） |
| **実装計画** | 何をどの順で実装するか（フェーズ・依存・リスク） |

---

## 2. スコープ

### 2.1 対象（本書で扱う）

1. **ダッシュボード／管理 UI**（既存 API の運用者向け画面）
2. **本番運用・デプロイ**（環境変数、Cron、LINE Developers、マイグレーション）
3. **データ整合性の検証**（OAM `customer_id` と Messaging `line_user_id`）
4. **品質基盤**（Lint 修復、最低限の E2E／API テスト方針）
5. **予約（Phase H）の拡張**（MVP からの段階的拡張：承認・カレンダー等はオプション扱いで定義）

### 2.2 対象外

- **I3**：外部 CRM／スプレッドシート連携（別紙または需要発生後に定義）
- LSTEP の**機能完全コピー**（当初計画どおり非目標）

# 第一部：仕様書

## 3. 機能要件

### 3.1 管理ダッシュボード（LINE サービス単位）

**目的**: API のみに頼らず、運用者がブラウザから主要操作ができること。

| ID | 機能 | 最低限の画面・操作 |
|----|------|-------------------|
| UI-1 | 接続状態 | Channel 認証が設定済みか、Webhook URL のコピー用表示 |
| UI-2 | コンタクト | 一覧・検索（タグ・属性フィルタは段階的）・詳細・タグ／属性編集 |
| UI-3 | タグ・属性・セグメント | CRUD、セグメントプレビュー人数の表示 |
| UI-4 | テンプレ・一斉配信 | テンプレ作成、配信ジョブ作成（対象ソース選択）、予約日時、ジョブ一覧・詳細・キャンセル／再試行 |
| UI-5 | MA ルール・シナリオ・リマインダ | 一覧・ON/OFF・作成／編集（JSON 直書きは避け、フォーム中心） |
| UI-6 | フォーム | フォーム・設問編集、セッション発行、回答一覧、公開 URL の表示 |
| UI-7 | リッチメニュー・postback | メニュー作成フロー（エリアは JSON エディタ or ウィザードのどちらかを明記）、バインディング一覧 |
| UI-8 | Flex テンプレ | 一覧・JSON 編集・プレビュー送信（テストユーザー指定） |
| UI-9 | 分析 | 短縮リンク一覧・クリック集計、CV レポート、OAM×セグメントレポート、ファネルレポートの閲覧 |
| UI-10 | 予約 | 予約サービス・スロット管理、予約一覧、キャンセル |
| UI-11 | 外部連携 | Outbound Webhook 設定、外部 API キー発行・失効（平文は一度だけの注意表示） |

**受け入れ基準（ダッシュボード）**

- 上表の **UI-1〜UI-6** を満たせば **MVP 合格**（残りは次イテレーション可）。
- 認証済みユーザーが **自分が権限を持つプロジェクト／サービス**のみ操作できること（既存の認証・RLS 方針に合わせる）。

### 3.2 本番運用・デプロイ

| ID | 要件 | 受け入れ基準 |
|----|------|----------------|
| OPS-1 | 環境変数一覧と説明 | `README` または `docs/` に **必須／任意**を表形式で記載 |
| OPS-2 | Supabase マイグレーション | 本番適用手順（`050`〜`057` など）が文書化され、CI または手順で再現可能 |
| OPS-3 | Cron | `line-messaging-broadcast`、`line-messaging-ma-scheduler` 等の **推奨間隔**と認証ヘッダを文書化 |
| OPS-4 | LINE Developers | Webhook URL、チャネルアクセストークン再発行時の手順 |

### 3.3 データ整合性（OAM × Messaging）

| ID | 要件 | 受け入れ基準 |
|----|------|----------------|
| DATA-1 | 突合レポートの運用 | 既存 API `oam-contact-reconcile` の結果を **定期確認する運用**が文書化される |
| DATA-2 | 不一致時の方針 | 不一致が一定率を超えた場合の **エスカレーション／マッピングテーブル検討**を「次アクション」として記載（実装は任意） |

### 3.4 品質基盤

| ID | 要件 | 受け入れ基準 |
|----|------|----------------|
| QA-1 | Lint | `npm run lint` が **CI で通る**（eslint-config-next の解決含む） |
| QA-2 | テスト | クリティカルパス（Webhook 署名、短縮リダイレクト、予約 RPC の失敗パターン等）に **最低 1 本ずつ**テストがある（方針を文書化） |

### 3.5 予約（Phase H）の拡張（オプション）

以下は **必須ではなく**、事業要件が出たらピックアップする。

| ID | 内容 |
|----|------|
| H-OPT-1 | 店舗・スタッフ・部屋などのリソースモデル |
| H-OPT-2 | 承認フロー（pending → confirmed） |
| H-OPT-3 | 外部カレンダー（Google Calendar 等）との同期 |
| H-OPT-4 | リマインド（予約 N 日前に Messaging 送信） |

---

## 4. 非機能要件

| 項目 | 内容 |
|------|------|
| セキュリティ | 外部 API キーはハッシュのみ保管、Outbound は HTTPS 推奨、シークレットのローテ手順 |
| 監査 | 重要操作（API キー発行・失効、Webhook URL 変更）をログまたはイベントで追えること（最低限は既存 `line_messaging_events` の活用を検討） |
| パフォーマンス | ダッシュボード一覧はページネーション必須。ファネル集計はコンタクト数増に応じた **バッチ集計 or 物化**を将来検討 |

# 第二部：実装計画

## 5. フェーズ構成（推奨順）

依存の浅い順に **UI 基盤 → 運用文書・Lint → データ検証 → 予約オプション** とする。

| フェーズ | 名称 | 主な成果物 | 依存 |
|----------|------|------------|------|
| **P1** | ダッシュボード MVP | UI-1〜UI-6 の画面、既存 `(dashboard)` 配下へのルーティング、LINE サービス専用ナビ | 既存認証 |
| **P2** | ダッシュボード拡張 | UI-7〜UI-11 | P1 |
| **P3** | 運用・品質 | OPS 文書、Lint 修復、テスト方針とクリティカルテスト追加 | — |
| **P4** | データ整合 | DATA-1/2 の運用ドキュメント、必要ならマッピング設計のスパイク | OAM 実データ |
| **P5** | 予約オプション | H-OPT のうち事業が選んだ項目のみ | P1 の予約画面 |

**I3 は全フェーズから除外する。**

## 6. マイルストーン（リリース単位）

| マイルストーン | 含む内容 | ユーザー価値 |
|----------------|----------|----------------|
| **R1** | P1 完了 | 運用者がブラウザでコンタクト・配信・MA・フォームの核を操作できる |
| **R2** | P2 完了 | リッチメニュー・分析・外部連携設定まで画面から完結 |
| **R3** | P3 完了 | CI が緑、オンボーディング文書で本番立ち上げ可能 |
| **R4** | P4 完了（運用中心） | OAM と Messaging の ID 整合が運用で可視化・判断できる |
| **R5** | P5（任意） | 業種特化の予約要件に対応 |

## 7. リスクと緩和

| リスク | 緩和 |
|--------|------|
| ダッシュボードのスコープ膨張 | UI-1〜6 を R1 の必須、7〜11 を R2 に固定 |
| Lint が環境依存で長引く | 再現用の Node バージョンを `package.json` / CI で固定 |
| ファネル集計の負荷 | コンタクト数しきい値を超えたらバッチ集計チケットを切る（本書では「将来検討」） |

## 8. 次アクション（着手順）

1. **P1 の画面一覧ワイヤ**（1 枚）と、既存 `projects/.../services/.../line` との統合方針を決める。  
2. **OPS-1 の環境変数表**を `docs/` または `README` に追記する（実装と並行可）。  
3. **QA-1**：eslint の失敗原因を特定し、`eslint.config` または依存バージョンを修正する PR を優先する。  
4. P1 実装 → R1 リリース → P2 へ。
