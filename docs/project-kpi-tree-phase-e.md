# KPIツリー・分析 Phase E 仕様メモ

> **位置づけ**: リポジトリ内の「Phase 1E（分析画面・グラフ）」実装計画書とは**別物**。ここでは **プロジェクト横断サマリー** を前提にした **KPIツリー DnD・プリセット・相関／回帰 MVP** を定義する。  
> **状態**: 未実装。実装時は本ドキュメントとマイグレーション `024`〜`027` を参照すること。

---

## 1. 目的

- プロジェクト単位で、**複数チャネル＋外生変数**を横断したデータを、**分析の問い（枝）**に沿って扱う。
- **制約付きツリー**で KPI の親子構造を保存し、**枝（プリセット）**ごとに **目的変数 Y** と **説明変数 X の集合**を切り替える。
- **相関・回帰の MVP** で、社内での仮説検討に使える最小機能を提供する。

---

## 2. 画面の置き場所

| 項目 | 方針 |
|------|------|
| **KPIツリー・プリセット・相関／回帰** | **プロジェクト横断サマリー** コンテキストに置く。 |
| **推奨 UI** | 横断サマリー画面内の **「分析」タブ**（レポート／データ表に続く）、または `/projects/[projectId]/analysis` などプロジェクト直下の専用ルート。 |
| **個別サービスのサマリー画面** | KPIツリー本体は**置かない**（将来、横断分析へのリンクのみ可）。 |

**理由**: ツリーの「枝＝新規／リピート／全体」等は、**複数サービスの列**とセットで意味を持つ。サービス単体画面ではデータと役割が一致しにくい。

## 3. データソース（既存）

実装済み／計画済みのテーブルと役割。

| マイグレーション | テーブル等 | 役割 |
|------------------|------------|------|
| `025_project_metrics_daily.sql` | `project_metrics_daily` | 日次キャッシュ。`(project_id, service_id, date, metric_ref)` → `value`。分析用ワイド表は **metric_ref を列にピボット**したイメージ。 |
| `026_project_summary_templates.sql` | `project_summary_templates` | プロジェクト横断テンプレート。`rows` に `metricRef` 等。表示・列選択のマスタ。 |
| `024_unified_summary_external.sql` | `project_external_daily` | 外生（祝日・天気等）。`project_id + date`。**説明変数候補**。 |
| `024` | `projects.latitude/longitude/location_name` | 天気 API 等の参照用。 |
| `027_service_custom_metrics.sql` | `service_custom_metrics` | サービス単位カスタム指標（`formula`）。テンプレ・集計パイプラインと組み合わせて列化。 |

**方針（プロダクト合意）**

- 指標は **テンプレートに載るカード** および **カスタム指標** を横断サマリーで利用する（詳細は横断サマリー実装に従う）。
- プロジェクト内の**アクティブなサービスのみ**対象。**論理削除済みサービス**は含めない。

## 4. KPIツリー（未実装）

### 4.1 挙動

- **制約付き DnD**（Miro 的な自由配置はしない）。
- 許可する操作の例: 同一階層での **並べ替え**、**別親への移動**（ループなし）。
- ノード種別の例: **フォルダ（中間）** / **リーフ（KPI・指標参照）**（実装時に enum 化してよい）。

### 4.2 保存（案・マイグレーション未作成）

新規テーブル案（名前は実装時に調整可）:

- `project_kpi_tree_nodes`  
  - `id`, `project_id`, `parent_id` (nullable), `sort_order`, `label`, `node_type`, `metric_ref` (nullable, リーフでテンプレ／カスタムと紐づくキー), `created_at`, `updated_at`

または JSON でツリー全体を `project_kpi_trees` に持つ方式。**正規化 vs JSON** はクエリ・バージョン管理のトレードオフで選択。

### 4.3 プリセット（枝 → Y / X）

- **プリセット**は `project_id` 単位で複数保持可。
- 各プリセットに少なくとも:
  - **名前**（例: 「新規獲得」「リピート」「全体」）
  - **目的変数**: 1 列（`metric_ref` またはワイド表の列キーと同一の識別子）
  - **説明変数**: 複数列（同上）
- UI は **プリセット選択で Y/X が自動切替**を基本とし、**上級者向けに列ピッカー**は後からでも可。

テーブル案:

- `project_analysis_presets`  
  - `id`, `project_id`, `name`, `target_metric_ref` (text), `feature_metric_refs` (text[] or jsonb), `created_at`, `updated_at`

ツリーの「枝」とプリセットを **1:1 で紐づける**か、**プリセットだけ独立リスト**にするかは実装時に決定（初期は独立リストでも可）。

## 5. 相関・回帰 MVP（未実装）

### 5.1 対象データ

- **入力**: 指定期間・粒度（日／週／月）で整えた **ワイド表**（1 行＝1 期間）。
- **列の源泉**: 横断テンプレート＋カスタム指標＋外生（`project_external_daily` の数値・フラグ列）のうち、**数値として解釈できる列のみ**。

### 5.2 相関

- ユーザーまたはプリセットが指定した **複数列**について、**ペアワイズ相関**（または相関行列）を表示。
- **欠損**: ペアごとに有効な観測だけで計算、または行方向リストワイズ削除 — 実装時に統一し、UI に短く注記。

### 5.3 回帰（MVP）

- **線形回帰 1 本**（または単純な正則化）でよい。
- **Y は 1 つ**、**X は複数**（定数項あり）。
- **観測数チェック**: 閾値未満（例: 有効日数 &lt; 30）は実行せず警告表示（閾値は定数または設定化）。

### 5.4 解釈・表示

- **相関 ≠ 因果** を短い注記で表示（社内向けでも最小限は推奨）。
- 係数・R²（あれば）を表または簡易テキストで表示。クライアント向けの**長文免責**は次期フェーズ（本ドキュメント §7）。

### 5.5 非スコープ（MVP）

- MMM、増分（Lift）実験設計、ラグ自動最適化、ベイズ回帰などは含めない。

## 6. 認証・権限

| 項目 | Phase E（本仕様） |
|------|------------------|
| **匿名アクセス** | **不可**。既存アプリと同様 **ログイン済み（Supabase `authenticated`）** のみ。 |
| **RLS** | 新規テーブルも `025`/`026` と同様、`authenticated` 全アクセスまたはプロジェクトメンバー制限は**次期**で整理。 |
| **ロール別（編集者のみツリー編集など）** | **次期**。MVP は全認証ユーザー同一権限でも可。 |
