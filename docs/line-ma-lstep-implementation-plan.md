# LINE MA（LSTEP 相当）実装計画

本書は、本リポジトリにおいて **LINE 公式アカウント向け MA 機能**を LSTEP 相当の水準まで拡張するための**実装計画**である。コードではなく、フェーズ・マイルストーン・依存関係・リスクを整理する。

---

## 1. 前提の固定

| 項目 | 方針 |
|------|------|
| プロダクト境界 | **1 サービス（LINE）・2 モジュール**（**OAM＝分析取り込み** / **Messaging＝ MA**） |
| データ統合 | **`service_id` を共有**。ユーザー突合は `line_messaging_contacts.line_user_id` と `line_oam_rewardcard_txns.customer_id`（OAM CSV の Customer ID）を同一文字列として比較する想定。**実データでの一致検証は継続タスク**。 |
| 中心モデル | **Trigger → Condition → Action**（シナリオ・自動応答・フォーム後処理・リマインダを同一エンジンに載せる設計を目指す） |
| 現状（本書執筆時点） | OAM バッチ・リワード付与ログ・`line_messaging_contacts`（マイグレーション 049）の**土台**。Messaging Webhook・配信エンジン・ CRM UI は**未着手または未完了**。 |

「LSTEP 同等」は**機能の完全コピー**ではなく、**同カテゴリで競合可能な水準**を目標とし、**MVP → 拡張**で切る前提とする。

---

## 2. ロードマップ全体（フェーズ）

依存の大枠は **A → B → C → D** がクリティカルパス。E〜I は並列化・前倒しの余地あり。

| Phase | 名称 | 概要 |
|-------|------|------|
| **A** | 基盤・接続 | Messaging API と DB を結び、ユーザー行を確実に蓄積 |
| **B** | 配信基盤 | テンプレ・一斉・予約・履歴・レート制限 |
| **C** | CRM・セグメント | タグ・属性・セグメント・絞り込み配信 |
| **D** | 自動化・ MA | イベントログ・アクションエンジン・自動応答・シナリオ・リマインダ |
| **E** | フォーム・流入 | フォーム定義・回答紐付け・回答後アクション・流入パラメータ |
| **F** | UI 導線強化 | リッチメニュー・出し分け・Flex / カルーセル |
| **G** | 分析・計測 | クリック・CV・ファネル・配信×OAM 来店分析・スコアリング |
| **H** | 予約 | 業種依存が強い。必要なら E〜F と並行で前倒し可 |
| **I** | 外部連携 | 公開 API・Outbound Webhook・外部 CRM 等 |

---

## 3. フェーズ別：目的・成果物・依存

### Phase A — 基盤・接続（最優先）

**目的**: Messaging API と DB を結び、ユーザー行を確実に蓄積する。

| ID | 内容 |
|----|------|
| A1 | **Channel 認証情報**の保存（`service_id` 単位：channel secret / long-lived token 等。暗号化方針を決める） |
| A2 | **Webhook エンドポイント**（署名検証、冪等、イベント種別のパース） |
| A3 | **`line_messaging_contacts` の UPSERT** の本番運用（follow / message / unfollow 等） |
| A4 | **配信 API クライアント**（push / multicast / reply の薄いラッパ） |
| A5 | **運用**: 開発用トンネル、本番 URL、LINE Developers 設定の手順書 |

**完了の定義**: 任意ユーザーにテスト push ができ、DB に `line_user_id` が残る。

---

### Phase B — 配信基盤

**目的**: 「絞って送る」前に **送れる・予約できる・履歴が残る**。

| ID | 内容 |
|----|------|
| B1 | **メッセージテンプレート**（テキスト中心。変数プレースホルダは段階的に拡張） |
| B2 | **一斉配信ジョブ**（対象リスト・送信状態・エラー・再試行） |
| B3 | **配信予約**（スケジューラ＋ジョブキュー。既存バッチとの役割分担を明文化） |
| B4 | **配信履歴**（誰に・いつ・何を・成功/失敗） |
| B5 | **レート制限・バッチ分割**（LINE の上限に合わせたチャンク送信） |

**完了の定義**: 運用者が UI から一斉配信・予約配信し、履歴を追える。

---

### Phase C — CRM・セグメント

**目的**: 友だち・タグ・セグメントの核。

| ID | 内容 |
|----|------|
| C1 | **タグ**（サービススコープ、`line_messaging_contacts` への多対多） |
| C2 | **カスタム属性**（定義テーブル＋値。型・選択肢） |
| C3 | **対応ステータス / 担当 / メモ**（運用向け最小） |
| C4 | **セグメント定義**（最初は「タグ AND/OR」「属性条件」程度のビルダー） |
| C5 | **セグメント配信**（Phase B のジョブにセグメント ID を渡す） |

**完了の定義**: タグで絞ったユーザーに一斉配信できる。OAM 側 `customer_id` との **突合検証レポート**を 1 本出せると安心。

---

### Phase D — 自動化・ MA（本丸）

**目的**: シナリオ・自動応答・リマインダを **同一アクションエンジン**に載せる。

| ID | 内容 |
|----|------|
| D1 | **イベントログ**（標準化された trigger 種別・payload。分析の前提） |
| D2 | **アクションエンジン v1**（タグ付与・属性更新・シナリオ開始・配信キュー投入） |
| D3 | **自動応答**（キーワード / 完全一致・部分一致。ルール優先度） |
| D4 | **シナリオ**（ステップ・待機・分岐・停止。状態は DB で保持） |
| D5 | **リマインダ**（日時トリガー＋対象 Contact） |
| D6 | **管理画面**（ルール一覧・ON/OFF・簡易デバッグ） |

**完了の定義**: 「友だち追加でタグ」「キーワードで返信」「N 日後にメッセージ」が再現できる。

---

### Phase E — フォーム・流入

| ID | 内容 |
|----|------|
| E1 | フォーム定義・設問タイプ・公開 URL |
| E2 | 回答保存（`line_user_id` またはセッション経由の紐付け） |
| E3 | 回答後アクション（タグ・シナリオ・配信） |
| E4 | 流入パラメータ（utm / line 導線 ID） |

---

### Phase F — UI 導線強化

| ID | 内容 |
|----|------|
| F1 | リッチメニュー（画像・エリア・postback） |
| F2 | 出し分け（タグ・属性に応じたメニュー ID） |
| F3 | Flex / カルーセル（テンプレ＋プレビュー） |

---

### Phase G — 分析・計測

| ID | 内容 |
|----|------|
| G1 | 短縮 URL / クリック計測（メッセージ内リンク） |
| G2 | コンバージョン定義 |
| G3 | **配信 × OAM 来店**（セグメント別付与率・期間ウィンドウ。`line_oam_rewardcard_txns` 活用） |
| G4 | ファネル・スコアリング（イベント辞書が揃ってから） |

**注意**: イベント設計は **Phase D 前後で辞書化**しておくと後戻りが減る。

---

### Phase H — 予約

業種優先なら **E〜F と並行または前倒し**。カレンダー・承認・リマインドはドメインが重いため **サブフェーズ分割**を推奨。

---

### Phase I — 外部連携

| ID | 内容 |
|----|------|
| I1 | Outbound Webhook（イベント通知） |
| I2 | API キー・スコープ（contacts / tags / 配信の最小 READ/WRITE） |
| I3 | 外部 CRM / スプレッドシートは需要に応じて |

## 4. マイルストーン（リリース単位）

| マイルストーン | 含めるフェーズ | ユーザー価値（一言） |
|----------------|----------------|----------------------|
| **M1** | A + B | 公式アカウントに「送れる・予約できる」 |
| **M2** | C | 「タグで絞って送れる」＋ OAM 突合の土台 |
| **M3** | D | 「反応に応じて動く」MA |
| **M4** | E + G（一部） | フォームで取り込み、配信〜来店を見える化 |
| **M5** | F + G 拡張 | 導線と分析で改善サイクル |
| **M6** | H + I | 予約・外部連携でエンタープライズ寄り |

---

## 5. 横断タスク（全フェーズで効く）

- **セキュリティ**: Webhook 署名、トークン保管、監査ログ、権限（クライアント／プロジェクト／サービス）
- **信頼性**: ジョブの再実行、デッドレター、OAM バッチとの **データ鮮度の説明**
- **非機能**: LINE レート制限、大量配信時のキュー、観測（ログ・メトリクス）

---

## 6. リスクと意思決定メモ

1. **`Customer ID` と `userId` の一致** — 不一致なら **マッピングテーブル**または **別キー取得経路**が必要。計画に **検証スプリント**を常設する。
2. **スコープ膨張** — 「LSTEP 相当」を **M1〜M3 で何を必須とするか** を別紙で固定するとブレない。
3. **リポジトリ境界** — Instagram 等既存機能と **同一ダッシュボードか別プロダクトか** で認証・ナビ・デプロイ境界を一度決める。
