# Instagram分析システム バッチ詳細仕様書

**バージョン**: 1.0
**作成日**: 2026-04-03
**実行基盤**: Supabase pg_cron + Supabase Edge Functions

---

## 1. バッチ実行基盤の構成

```
pg_cron（スケジュール管理）
        ↓
Supabase Edge Functions（実際の処理）
        ↓
Instagram Graph API
        ↓
Supabase PostgreSQL（データ保存）
        ↓
Claude API（AI分析バッチのみ）
```

### 実行基盤の選定理由

| 課題 | 対策 |
|------|------|
| Vercelのタイムアウト（最大300秒）| Edge Functionsで処理（最大150秒）、キューで分割 |
| APIレート制限（200コール/時） | アカウントごとにキューイングして順次処理 |
| バッチ失敗時のリカバリ | `batch_job_logs` で状態管理、手動再実行対応 |
| Webhook追加時の対応 | Webhook受信エンドポイントを別途追加するだけで既存設計に影響なし |

## 3. 各バッチ詳細仕様

### 3.1 `token_refresh_batch` — アクセストークン更新

**目的**: Long-Lived Access Token（有効期限60日）を定期的に更新して期限切れを防ぐ

**実行タイミング**: 毎日 AM 2:00

**処理フロー**:
```
1. ig_account_tokens から有効なトークンを全件取得
2. expires_at が 20日以内のトークンを更新対象とする
3. Meta Graph API の token refresh エンドポイントを呼び出す
   GET /oauth/access_token
     ?grant_type=ig_refresh_token
     &access_token={current_token}
4. 新しいトークンと expires_at を ig_account_tokens に上書き保存
5. 更新失敗時はアカウントの status を 'disconnected' に変更
6. バッチログに結果を記録
```

**API仕様**:
- エンドポイント: `GET /oauth/access_token`
- パラメータ: `grant_type=ig_refresh_token`, `access_token`
- 更新後の有効期間: 60日

**エラーハンドリング**:
- トークン無効（400エラー）→ アカウントを `disconnected` に変更、管理画面でアラート表示
- 通信エラー → 3回リトライ後、スキップして次のアカウントへ

### 3.2 `account_sync_batch` — アカウント基本情報同期

**目的**: アカウントのプロフィール情報（フォロワー数含む）を最新化する

**実行タイミング**: 毎日 AM 3:00

**処理フロー**:
```
1. status が 'active' の全アカウントを取得
2. アカウントごとに以下のAPIを呼び出す
   GET /{ig-user-id}
     ?fields=id,username,name,biography,followers_count,follows_count,
             media_count,profile_picture_url,website
3. ig_accounts テーブルを更新（last_synced_at も更新）
4. followers_count の変化を ig_account_insight_fact にスナップショット保存
5. バッチログに記録
```

**取得フィールド**:

| フィールド | 保存先 |
|-----------|-------|
| id | platform_account_id |
| username | username |
| name | account_name |
| biography | biography |
| followers_count | followers_count + ig_account_insight_fact |
| follows_count | follows_count |
| media_count | media_count |
| profile_picture_url | profile_picture_url |
| website | website |

### 3.3 `media_index_batch` — 投稿一覧取得・更新

**目的**: 新規投稿の検知と投稿マスタの最新化

**実行タイミング**: 毎時 0分

**処理フロー**:
```
1. status が 'active' の全アカウントを取得
2. アカウントごとに以下のAPIを呼び出す
   GET /{ig-user-id}/media
     ?fields=id,caption,media_type,media_product_type,media_url,
             thumbnail_url,permalink,timestamp,is_comment_enabled,
             shortcode,children{id,media_url,thumbnail_url}
     &limit=50
3. 取得した投稿一覧と ig_media を照合
   - 新規投稿 → INSERT
   - 既存投稿でキャプション変更など → UPDATE
   - 削除済み投稿 → is_deleted = true に UPDATE
4. ストーリー（media_product_type = 'STORY'）は別フラグで管理
5. カルーセル投稿の子メディアは children_json に保存
6. APIレート使用量（X-App-Usage）を api_call_logs に記録
7. バッチログに記録
```

**ページネーション処理**:
- `after` カーソルを使って最大200件まで取得
- 2回目以降のバッチでは `since` パラメータで差分取得を試みる
- 初回取得時のみ全件取得

### 3.4 `media_insight_batch` — 投稿Insight取得（優先度別）

**目的**: 投稿ごとの指標値を時系列で収集する

**実行タイミング**: 毎時 10分（media_index_batch の後）

**優先度別の取得ウィンドウ**:

| 優先度 | 対象投稿 | 実行頻度 |
|-------|---------|---------|
| 高（Tier1） | 投稿後48時間以内 | 毎時実行 |
| 中（Tier2） | 投稿後3〜7日 | 6時間おき |
| 低（Tier3） | 投稿後8〜30日 | 1日1回 |
| 保管（Tier4） | 投稿後31日以降 | 週1回 |

**処理フロー**:
```
1. 対象Tierの投稿リストを ig_media から取得
2. アカウントのレート使用量を api_call_logs から確認
   → 使用量が80%を超えていればスキップ（次の時間帯に延期）
3. 各投稿に対して以下のAPIを呼び出す

   [FEED / CAROUSEL の場合]
   GET /{media-id}/insights
     ?metric=impressions,reach,likes,comments,shares,saved,
             profile_visits,follows,total_interactions
     &period=lifetime

   [REELS の場合]
   GET /{media-id}/insights
     ?metric=impressions,reach,likes,comments,shares,saved,
             ig_reels_video_view_total_time,
             ig_reels_aggregated_all_plays_count,
             total_interactions
     &period=lifetime

   [VIDEO の場合]
   GET /{media-id}/insights
     ?metric=impressions,reach,likes,comments,shares,saved,
             video_views,total_interactions
     &period=lifetime

4. 取得結果を ig_media_insight_fact にスナップショットとして INSERT
5. ig_media_snapshot も同時に更新（like_count, comments_count）
6. X-App-Usage を確認し、閾値超過時は処理を中断
7. バッチログに記録
```

**取得指標一覧**:

| metric_code | FEED | REELS | VIDEO | STORY |
|------------|------|-------|-------|-------|
| impressions | ✅ | ✅ | ✅ | ✅ |
| reach | ✅ | ✅ | ✅ | ✅ |
| likes | ✅ | ✅ | ✅ | - |
| comments | ✅ | ✅ | ✅ | - |
| shares | ✅ | ✅ | ✅ | ✅ |
| saved | ✅ | ✅ | ✅ | - |
| video_views | - | - | ✅ | ✅ |
| ig_reels_aggregated_all_plays_count | - | ✅ | - | - |
| total_interactions | ✅ | ✅ | ✅ | - |
| profile_visits | ✅ | ✅ | - | - |
| follows | ✅ | ✅ | - | - |

**レート制限管理**:
```javascript
// X-App-Usage ヘッダの例
// {"call_count": 15, "total_time": 8, "total_cputime": 3}
// → call_count が 80 を超えたら処理を中断

```

### 3.8 `kpi_calc_batch` — KPI計算

**目的**: Fact層のデータをもとにKPIを計算してkpi_resultに保存する

**実行タイミング**: 毎時 30分（media_insight_batch の後）

**処理フロー**:
```
1. kpi_master から is_active = true の全KPIを取得
2. KPIを capability_type で分類
3. DIRECT_API タイプ: ig_*_insight_fact から最新値を取得してそのまま保存
4. DERIVED タイプ: formula_expression に従って計算
5. 計算結果を kpi_result に UPSERT
   - source_status を適切に設定
     * 依存データが揃っている → 'complete'
     * 一部データのみ → 'partial'
     * データ未取得 → 'waiting_data'
6. バッチログに記録
```

**KPIタイプ別の計算方法**:

#### Type A: DIRECT_API（直接取得）
```sql
-- 例: reach_per_post
INSERT INTO kpi_result (account_id, media_id, kpi_id, grain, actual_value, ...)
SELECT
  m.account_id,
  f.media_id,
  k.id as kpi_id,
  'lifetime' as grain,
  f.value as actual_value,
  'complete' as source_status,
  now() as calculated_at
FROM ig_media_insight_fact f
JOIN ig_media m ON f.media_id = m.id
JOIN kpi_master k ON k.kpi_code = 'reach_per_post'
WHERE f.metric_code = 'reach'
  AND f.snapshot_at = (
    SELECT MAX(snapshot_at) FROM ig_media_insight_fact
    WHERE media_id = f.media_id AND metric_code = 'reach'
  );
```

#### Type B: DERIVED（算出）
```sql
-- 例: engagement_rate = (likes + comments + saves + shares) / reach * 100
WITH latest_metrics AS (
  SELECT
    media_id,
    MAX(CASE WHEN metric_code = 'likes' THEN value END) as likes,
    MAX(CASE WHEN metric_code = 'comments' THEN value END) as comments,
    MAX(CASE WHEN metric_code = 'saved' THEN value END) as saves,
    MAX(CASE WHEN metric_code = 'shares' THEN value END) as shares,
    MAX(CASE WHEN metric_code = 'reach' THEN value END) as reach
  FROM ig_media_insight_fact
  WHERE snapshot_at > now() - interval '2 hours'
  GROUP BY media_id
)
SELECT
  m.account_id,
  lm.media_id,
  k.id as kpi_id,
  CASE WHEN lm.reach > 0
    THEN (COALESCE(lm.likes,0) + COALESCE(lm.comments,0) +
          COALESCE(lm.saves,0) + COALESCE(lm.shares,0))
         / lm.reach * 100
    ELSE NULL
  END as actual_value
FROM latest_metrics lm
JOIN ig_media m ON lm.media_id = m.id
JOIN kpi_master k ON k.kpi_code = 'engagement_rate';
```

#### Type C: 時系列（初速指数の例）
```sql
-- 早速指数: 投稿後1時間のreachを、過去30日の平均初速と比較した比率
WITH hourly_reach AS (
  SELECT
    media_id,
    value as reach_1h
  FROM ig_media_insight_fact
  WHERE metric_code = 'reach'
    AND snapshot_at BETWEEN posted_at AND posted_at + interval '1 hour' + interval '10 min'
),
avg_reach AS (
  SELECT
    m.account_id,
    AVG(hr.reach_1h) as avg_reach_1h
  FROM hourly_reach hr
  JOIN ig_media m ON hr.media_id = m.id
  WHERE m.posted_at > now() - interval '30 days'
  GROUP BY m.account_id
)
SELECT
  m.account_id,
  hr.media_id,
  hr.reach_1h / NULLIF(ar.avg_reach_1h, 0) as early_velocity_index
FROM hourly_reach hr
JOIN ig_media m ON hr.media_id = m.id
JOIN avg_reach ar ON m.account_id = ar.account_id;
```

### 3.9 `target_compare_batch` — 目標値比較・進捗更新

**目的**: KPI実績と目標値を比較してkpi_progressを更新する

**実行タイミング**: 毎日 AM 5:00

**処理フロー**:
```
1. 有効なkpi_targetを全件取得
2. 各targetに対応するkpi_resultの最新値を取得
3. 達成率を計算: achievement_rate = actual_value / target_value * 100
4. ステータスを判定:
   - achievement_rate >= 100 → 'achieved'
   - warning_threshold 以上 → 'on_track'
   - critical_threshold 以上 → 'warning'
   - critical_threshold 未満 → 'critical'
   - actual_value が NULL → 'insufficient_data'
5. kpi_progress に INSERT
6. バッチログに記録
```

### 3.12 `backfill_batch` — 過去データ再取得

## 4. レート制限管理の実装

### X-App-Usage の監視

```typescript
// Edge Function 内での実装イメージ
interface RateUsage {
  call_count: number;    // API呼び出し回数の使用率(%)
  total_time: number;    // 合計処理時間の使用率(%)
  total_cputime: number; // CPU時間の使用率(%)
}

async function checkRateLimit(headers: Headers): Promise<boolean> {
  const usage = JSON.parse(headers.get('X-App-Usage') || '{}') as RateUsage;

  // いずれかが80%を超えたら処理を中断
  const isOverLimit =
    (usage.call_count ?? 0) > 80 ||
    (usage.total_time ?? 0) > 80 ||
    (usage.total_cputime ?? 0) > 80;

  // レート使用量をDBに記録
  await saveRateUsage(usage);

  return isOverLimit;
}
```

### レート制限到達時の動作

```
1. 現在の処理を中断
2. batch_job_logs に status = 'partial' で記録
3. 未処理の投稿IDを metadata_json に保存
4. 次の実行時間（1時間後）まで待機
5. 次回実行時に metadata_json の未処理リストから再開
```

## 5. エラーハンドリング方針

### エラーの種類と対応

| エラー種別 | HTTP Status | 対応 |
|-----------|------------|------|
| トークン期限切れ | 400 (OAuthException) | アカウントを disconnected に変更 |
| レート制限 | 429 | 1時間待機後リトライ |
| 一時的なエラー | 500, 503 | 指数バックオフで3回リトライ |
| アカウント停止 | 400 (PermissionError) | アカウントを paused に変更 |
| ネットワークエラー | タイムアウト | 30秒待機後リトライ、3回で失敗 |

### リトライの実装

```typescript
async function withRetry<T>(
  fn: () => Promise<T>,
  maxRetries = 3,
  baseDelay = 1000
): Promise<T> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      if (attempt === maxRetries - 1) throw error;

      const delay = baseDelay * Math.pow(2, attempt); // 指数バックオフ
      await sleep(delay);
    }
  }
  throw new Error('Max retries exceeded');
}
```

## 8. Webhook対応時の追加設計（Phase2以降）

Webhookを追加する際は、以下のエンドポイントを追加するだけで既存設計への影響を最小化できます。

```
supabase/
  functions/
    webhook-receiver/          # Webhook受信エンドポイント
      index.ts                 # Meta からのイベントを受け取り、キューに積む
```

**Webhookで受け取れる主なイベント**:
- `mentions` — メンション
- `comments` — コメント
- `live_comments` — ライブコメント

Webhookイベント受信後は、対応するバッチ処理（comment_sync_batch等）を即時トリガーする設計にします。
